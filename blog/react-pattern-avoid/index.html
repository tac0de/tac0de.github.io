ReactJS - 피해야할 흔한 패턴

01 March, 2020 - 2 min read

리엑트는 사용자 인터페이스를 만들게 해주는 아주 강력한 라이브러리인 반면, 연약한 면들도 분명히 존재합니다.

대게 흔하게 마주치는 버그들 중에 DOM에 직접적인 조작을 리엑트와 같이 쓸 때 일어나는 것들이 있습니다.

이것은 정확히 패턴에 어긋나는 것은 아니지만 알맞은 상황에 따라서 전체적인 리엑트 애플리케이션, 프로젝트가 엉키고 꼬이고 망가질 수 있고, 전체적인 디버그 역시 매우 까다로워질 수 있습니다.

실질적인 문제 설명과 고치는 방법에 대해 들어가기 전에 우리가 버그를 어떻게 생성하는지 다음의 최소한의 코드 예제를 통해 알아봅시다.

const destroyElement = () =>
  document.getElementById("app").removeChild(document.getElementById("my-div"))

const App = () => {
  const [elementShown, updateElement] = React.useState(true)

  return (
    <div id="app">
      <button onClick={() => destroyElement()}>
        Delete element via querySelector
      </button>
      <button onClick={() => updateElement(!elementShown)}>
        Update element and state
      </button>
      {elementShown ? <div id="my-div">I am the element</div> : null}
    </div>
  )
}

ReactDOM.render(<App />, document.getElementById("root"))

위의 간단한 리엑트 예제는 컨테이너, 두 버튼과 state 변수가 있습니다.

destroyElement() 를 실행하는 버튼을 클릭하고 다른 버튼을 클릭하면 크래시가 뜰 것입니다.

왜일까요?

여기서의 문제가 즉각적으로 분명하지 않을 수 있지만, 브라우저 콘솔을 본다면 다음과 같은 예외 에러를 알아낼 수 있을 것입니다.

Uncaught DOMException: Failed to execute 'removeChild' on 'Node': The node to be removed is not a child of this node.

이해하기 힘들지도 모릅니다.

설명을 하자면, 리엑트는 자기들만의 DOM 을 대표하는 Virtual DOM 이라는 것을 사용하는데 무엇을 랜더할지 판단할 때 쓰입니다.

보통은 Virtual DOM 이 실제 DOM 과 비교를 통해 props, state 변경을 진행하는데 이는 Virtual DOM 을 업데이트 하고 실제 DOM 에 필수적인 변경사항을 전달해줍니다.

하지만 위와 같은 경우에는 destroyElement() 가 #my-div 를 지우기 떄문에 리엑트의 Virtual DOM 과 실제 DOM 이 다릅니다.

리엑트가 실제 DOM 를 Virtual DOM 의 변경사항을 통해 업데이트 하려고 할 때, #my-div 는 존재하지 않기 때문에 지워질 수가 없습니다.

이 예제는 destroyElement() 를 App 컴포넌트에 포함시키고 state 와 상호작용 함으로써 짧고 쉬운 해결을 할 수 있습니다.

하지만 이는 리엑트도 특정 상황에서 이러한 허점들이 분명히 존재 할 수 있음을 보여줍니다.

우리가 팀워크를 하면서 여러 개발자들이 서로 다른 일들을 하면서 코드를 공유하게 될 때 이러한 문재점들이 나올 수 있을 것입니다.

문재적 코드를 추적하는게 매우 까다로울 수 있으며 이 때문에 우리는 리엑트로 DOM 을 직접적으로 조작할때 매우 주의해야할 필요가 있습니다.
